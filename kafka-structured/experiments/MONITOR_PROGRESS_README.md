# Progress Monitoring Documentation

## Overview

The `monitor_progress()` function provides real-time monitoring of experiment suite execution by parsing the log file generated by `run_experiments.py`. It's designed to be non-blocking and can be called periodically (e.g., every 30 seconds) to track progress without interfering with the running experiments.

## Function Signature

```python
def monitor_progress(
    log_file_path: str,
    poll_interval_seconds: int = 30
) -> Optional[dict]:
    """
    Monitor experiment progress by parsing the log file.
    
    Args:
        log_file_path: Path to the experiment log file to monitor
        poll_interval_seconds: Polling interval (for display purposes only)
    
    Returns:
        Dictionary with progress information or None if log file doesn't exist
    """
```

## Return Value

When the log file exists and has content, returns a dictionary with:

```python
{
    'completed_runs': int,        # Number of completed runs
    'total_runs': int,            # Total number of runs (34)
    'current_run': str,           # Current run ID (e.g., "proactive_constant_run01")
    'current_condition': str,     # Current condition ("proactive" or "reactive")
    'current_pattern': str,       # Current load pattern ("constant", "sine", "spike", "step")
    'elapsed_minutes': float,     # Total elapsed time in minutes
    'eta': str,                   # Estimated time of completion (HH:MM:SS)
    'last_lines': List[str]       # Last 20 lines of log for display
}
```

Returns `None` if:
- Log file doesn't exist yet
- Log file is empty
- Log file cannot be read

## Usage Examples

### Example 1: Simple Polling Loop

```python
import time
from pipeline_orchestrator import monitor_progress, display_progress_dashboard

log_file = "kafka-structured/experiments/experiment_run_log.txt"

while True:
    progress_info = monitor_progress(log_file)
    display_progress_dashboard(progress_info)
    
    # Check if complete
    if progress_info and progress_info['completed_runs'] >= progress_info['total_runs']:
        print("All experiments completed!")
        break
    
    time.sleep(30)  # Poll every 30 seconds
```

### Example 2: Monitoring with Process Handle

```python
from pipeline_orchestrator import (
    launch_experiment_suite,
    monitor_progress,
    display_progress_dashboard
)

# Launch experiments
log_file = "kafka-structured/experiments/experiment_run_log.txt"
process = launch_experiment_suite(experiment_config, log_file)

# Monitor while process is running
try:
    while process.poll() is None:
        time.sleep(30)
        progress_info = monitor_progress(log_file)
        display_progress_dashboard(progress_info)
except KeyboardInterrupt:
    print("Monitoring stopped. Experiment continues in background.")

# Wait for completion
process.wait()
```

## Key Features

### Non-Blocking Design
- Function completes quickly (< 1 second)
- Reads only the last ~10KB of log file for efficiency
- Safe to call repeatedly without performance impact

### Robust Parsing
- Handles missing or incomplete log files gracefully
- Parses multiple log formats (PROGRESS markers, RUN headers, COMPLETED markers)
- Extracts most recent information by parsing in reverse order

### Progress Tracking
- Tracks completed runs vs total runs
- Identifies current run details (condition, pattern, run ID)
- Calculates elapsed time and ETA
- Estimates remaining time based on average run duration

### Error Handling
- Returns None for non-existent or empty files
- Handles malformed log lines gracefully
- Continues parsing even if some lines fail

## Implementation Details

### Log File Parsing Strategy

The function parses three types of markers in the log file:

1. **PROGRESS markers**: `PROGRESS: {idx}/{total} | Elapsed: {elapsed}min | ETA: {eta}`
   - Provides completed/total counts, elapsed time, and ETA

2. **RUN headers**: `#  RUN {idx}/{total}: {condition}_{pattern}_run{id}`
   - Identifies current run details

3. **COMPLETED markers**: `>>> COMPLETED {idx}/{total}: {label} <<<`
   - Confirms run completion for accurate counting

### Efficiency Optimizations

- Reads only last ~10KB of log file (not entire file)
- Keeps only last 50 lines in memory for parsing
- Parses in reverse order to find most recent information quickly
- Uses line buffering for file I/O

### Polling Interval

The recommended polling interval is **30 seconds**:
- Frequent enough for responsive updates
- Infrequent enough to avoid I/O overhead
- Matches the experiment's metric collection interval

## Troubleshooting

### Log file not found
- Ensure experiment suite has been launched
- Check log file path is correct
- Verify write permissions on log directory

### Progress shows 0/0 runs
- Log file may be in early initialization stage
- Wait for first PROGRESS marker to be written
- Check if run_experiments.py is actually running

### Elapsed time not updating
- Verify log file is being written to
- Check if experiment process is still running
- Ensure log file has write buffering enabled

### Dashboard shows "Unknown" values
- Log file may not have enough content yet
- Wait for first RUN header to be written
- Check log file format matches expected patterns

## Performance Considerations

- **Memory**: Function uses minimal memory (~50 lines × ~100 chars = ~5KB)
- **I/O**: Reads only last 10KB of log file per poll
- **CPU**: Parsing is fast (< 100ms for typical log files)
- **Disk**: No writes, only reads

Safe to run continuously for hours without performance degradation.
